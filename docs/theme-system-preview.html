How to keep this living:
1. Canonical palette source: js/packages/desktop-toolkit/src/theme/themeSystem.js (TOOLKIT_PALETTES). Add new palettes there first, then mirror into the PALETTES array in this file so the standalone preview stays in sync.
2. Keep token names stable to avoid regressions (bg, surface, text, muted, accent, accentText, border, success, warning, error, info).
3. Update this preview whenever toolkit theme behavior changes — consumer apps inherit it via @chamber-19/desktop-toolkit/theme.

App theme override pattern (CSS-only swap supported). Consumer apps inherit the shell theme by default (sidebar, topbar, and workspace foundation read from --ch-bg, --ch-surface, --ch-accent, etc.). To give an app a distinct palette, override the four palette tokens under a scoped attribute: 
[data-app-theme="transmittals"] {
  --ch-bg: #1A1C1F;       /* sidebar + topbar chrome */
  --ch-surface: #2A2D31;  /* drives content-bg + panel-bg */
  --ch-accent: #5C8EB8;
  --ch-border: #3B4148;
}
All derived tokens (--ch-content-bg, --ch-panel-bg, --ch-workspace-blend, --ch-active-bg, --ch-panel-border, etc.) are emitted as CSS expressions — color-mix(in srgb, var(--ch-bg) 88%, var(--ch-surface)) for solid blends and rgb(from var(--ch-accent) r g b / 0.16) for alpha overlays — so they recompute automatically when the palette tokens change. No JS hook needed. The toolkit's applyToolkitThemeVariables() JS API is still available for programmatic palette swaps (e.g. user choosing a palette from a picker), but a pure CSS attribute toggle is now sufficient for static app theming.

Three-zone topbar slots. The topbar grid (auto 1fr auto) exposes three zones: .scaffold-topbar-left (page title + optional sub-title), .scaffold-topbar-center (app context — breadcrumbs, project picker, filters), and .scaffold-topbar-right (app quick-actions + app status + shell theme + shell clock). Empty app slots collapse via :empty and render nothing. Apps fill the slots; the shell never injects fake placeholder UI.

Typography token contract (font-style axis is planned, not yet implemented). The shell exposes three semantic font tokens — --ch-font-display, --ch-font-ui, --ch-font-mono — emitted by resolveEdgeMeltVariables(). The preview consumes these via font-family: var(--ch-font-X) instead of hardcoding font names. Defaults today: 'Instrument Serif' / 'DM Sans' / 'JetBrains Mono' with system fallbacks, matching what the shell rendered before tokenization. Visual output is unchanged.

These font tokens are the contract for a future orthogonal typography axis, planned for a follow-up PR. The design intent is: 
<body data-app-theme="forge" data-font-style="lofi">
where data-app-theme controls color / surfaces / accents and data-font-style controls typography mood — the two axes are independent so an app can mix any palette with any typography preset. Planned font-style modes (final font choices TBD):
  • default — current toolkit stacks
  • technical — sharper, more system-feeling (candidates: Source Serif 4 / Manrope / IBM Plex Mono)
  • lofi — softer, editorial (candidates: Cormorant Garamond / Nunito Sans / IBM Plex Mono)
Status: data-font-style behavior is NOT implemented. No [data-font-style="..."] rules exist. Weight / leading / tracking tokens are not yet emitted — those will land alongside the actual font-style mode implementation. This PR establishes the family-token contract only so the future axis has clean plumbing.

Toolkit shell CSS (out of scope this PR): activation.css, updater.css, ReleaseNotes.css, and splash.css still carry hardcoded font-family declarations (legacy Inter usage in splash/updater). Their tokenization is queued as a separate typography sweep — touching them now would broaden scope beyond what this PR was scoped to.

Shell vs. content boundary. The toolkit shell owns the sidebar, topbar, workspace foundation, theme tokens, accent inheritance, and chrome→workspace blending — these are consistent across every consumer app. The content shown above (Overview / Recent Activity / System with metric cards) is the Dashboard app's internal layout, not a global template. Future apps — Transmittals, Drawings, Activation, Settings, project tools — fill the workspace with whatever rhythm fits: tables, queues, forms, file browsers, settings panels. The shell never forces dashboard styling. Reusable utilities (.section-heading, .section-paragraph, .content-divider) are opt-in.
